<a name="1"></a>
## 1. Самое главное за 60 секунд

Сайт — это **5 страниц**: Главная, Services (услуги), Process (процесс), About (о нас), Contact (контакты с формой).

Запомни три места, где ты будешь что‑то менять в 90% случаев:

| Хочешь поменять… | Иди сюда |
|---|---|
| **Тексты услуг, процесса, «о нас», опции формы** | папка [`content/`](content) |
| **Название компании, email, пункты меню, надписи на кнопках** | файл [`lib/site.ts`](lib/site.ts) |
| **Цвета, шрифты, скругления** | файл [`app/globals.css`](app/globals.css) |

Текст, который относится только к одной странице (например, заголовок героя на главной), лежит **прямо в файле этой страницы** внутри папки [`app/`](app).

## 2. Словарь терминов (читать первым)

Прежде чем нырять — короткий словарь. Я буду ссылаться на эти слова дальше.

- **Фреймворк (framework)** — готовый «каркас» с правилами и инструментами, на котором строят приложение, чтобы не писать всё с нуля. *Пример: наш сайт построен на фреймворке **Next.js**.*

- **Next.js** — фреймворк для сайтов на основе React. Его главная фишка для нас: **каждый файл `page.tsx` в папке `app/` автоматически становится отдельной страницей сайта** (об этом ниже). *Пример: файл `app/about/page.tsx` → страница `сайт.com/about`.*

- **React** — библиотека (готовый набор кода), которая позволяет строить интерфейс из **компонентов**. *Пример: кнопка, шапка, карточка услуги — всё это React‑компоненты.*

- **Компонент (component)** — переиспользуемый «кирпичик» интерфейса: кусок разметки + логика, у которого есть имя, и который можно вставлять много раз. *Пример: компонент `Button` (кнопка) используется и в шапке, и на главной, и в форме — описан один раз, применяется везде.*

- **Props (пропсы, «свойства»)** — настройки, которые передают компоненту снаружи, чтобы он отрисовался по‑разному. Это как аргументы функции. *Пример: `<Button variant="primary" size="lg">Book a Free Audit</Button>` — здесь `variant`, `size` и текст внутри — это пропсы.*

- **TSX / TypeScript (`.tsx`, `.ts`)** — язык программирования (TypeScript = JavaScript + проверка типов). Файлы `.tsx` содержат и логику, и разметку (HTML‑подобный код прямо в коде). *«Проверка типов» = редактор заранее ругается, если ты, например, передал число туда, где ждали текст.*

- **JSX/TSX‑разметка** — HTML‑подобные теги прямо внутри кода. *Пример: `<h1>Привет</h1>` внутри `.tsx` файла.*

- **Tailwind CSS** — способ задавать оформление (цвет, отступ, размер) короткими «классами» прямо в разметке, вместо отдельных CSS‑файлов. *Пример: `className="text-lg font-semibold text-gold"` = «крупный, полужирный, золотого цвета текст».*

- **CSS‑переменная / дизайн‑токен (design token)** — именованное значение цвета/размера, заданное в одном месте и используемое везде. Поменял в одном месте — поменялось на всём сайте. *Пример: `--gold` = золотой цвет; меняешь его раз — и золото меняется на всех кнопках и иконках.*

- **Сервер (server) / клиент (client)** — «сервер» = компьютер, который собирает страницу и отдаёт её; «клиент» = браузер пользователя. В Next.js большинство компонентов считаются на сервере (быстро, хорошо для поисковиков), а те, что должны реагировать на действия пользователя (клики, ввод), помечаются `"use client"` и работают в браузере.

- **`"use client"`** — пометка в самой первой строке файла‑компонента, означающая «этот кирпичик должен работать в браузере, потому что в нём есть интерактив» (клики, состояние, ввод). *Пример: форма, переключатель темы, мобильное меню — все с `"use client"`.*

- **Состояние (state)** — «память» компонента о том, что сейчас происходит. *Пример: открыто меню или закрыто; идёт ли отправка формы прямо сейчас.*

- **Импорт (import)** — строка вверху файла, которая «подключает» код из другого файла. *Пример: `import { Button } from "@/components/ui/button"` = «возьми кнопку из файла button».*

- **`@/` в путях импорта** — сокращение, означающее «от корня проекта». *Пример: `@/lib/site` = файл `lib/site.ts` в корне проекта. Настроено в `tsconfig.json`.*

- **Маршрут / роут (route)** — адрес страницы. *Пример: `/contact` — это роут страницы контактов.*

- **API‑роут (API route)** — серверный обработчик, который не показывает страницу, а принимает данные и что‑то с ними делает. *Пример: `app/api/contact/route.ts` принимает заполненную форму и отправляет письмо.*

- **Schema / валидация (Zod)** — описание «как должны выглядеть правильные данные» и автоматическая проверка. *Пример: «email обязателен и должен быть похож на email» — если нет, форма покажет ошибку.*

- **Переменная окружения (environment variable, env var)** — секретная или настроечная величина, которую задают отдельно от кода (чтобы не хранить пароли/ключи в коде). *Пример: `RESEND_API_KEY` — ключ почтового сервиса. Лежит в файле `.env.local`, который не попадает в git.*

- **SEO / метаданные (metadata)** — невидимая для глаз информация о странице для Google и соцсетей (заголовок вкладки, описание, картинка превью). *Пример: `metadata` в начале файла страницы.*

<a name="3"></a>
## 3. Из каких технологий собран сайт

Полный список — в файле [`package.json`](package.json) (это «список ингредиентов» проекта: какие библиотеки используются и какие команды можно запускать).

| Что | Зачем | Где «живёт» |
|---|---|---|
| **Next.js 14** | каркас сайта, маршруты, сервер | вся папка `app/` |
| **React 18** | компоненты интерфейса | везде в `.tsx` |
| **TypeScript** | язык + проверка типов | все `.ts`/`.tsx` |
| **Tailwind CSS** | оформление через классы | `className="..."` везде + `tailwind.config.ts` |
| **Radix UI** | готовая «начинка» аккордеона (раскрывающиеся блоки) с правильной доступностью | `components/ui/accordion.tsx` |
| **React Hook Form** | управление формой контактов | `components/contact/contact-form.tsx` |
| **Zod** | проверка данных формы | `lib/contact-schema.ts` |
| **Lucide** | иконки (значки) | импортируются по имени, напр. `MapPin`, `Mail` |
| **Шрифты Inter + Plus Jakarta Sans** | основной текст + заголовки | подключены в `app/layout.tsx` |

> 💡 Важно: **базы данных и «админки» (CMS) у сайта нет**. Это сделано осознанно для первой версии. Весь контент — это обычные текстовые файлы в коде (в папке `content/`). Чтобы поменять текст, меняют файл и заново публикуют сайт. Это нормально и просто для сайта‑визитки.

<a name="4"></a>
## 4. Как вообще работает сайт, когда кто‑то заходит

Представь путь пользователя, который открыл `harborlinesystems.com/about`:

```
┌──────────────┐   1. запрос адреса /about
│   Браузер    │ ───────────────────────────────►┐
│ (пользователь)│                                  │
└──────────────┘                                  ▼
                                       ┌────────────────────────┐
                                       │       Next.js          │
                                       │  (сервер собирает      │
                                       │   страницу)            │
                                       └────────────────────────┘
                                                  │
        2. берёт "обёртку" (общую для всех страниц):│
           app/layout.tsx → шапка + подвал + меню   │
                                                  │
        3. вставляет внутрь нужную страницу:        │
           app/about/page.tsx                      │
                                                  │
        4. страница берёт тексты из content/        │
           и кирпичики из components/               │
                                                  ▼
                                       ┌────────────────────────┐
                                       │  Готовый HTML + стили   │
                                       └────────────────────────┘
                                                  │ 5. отдаётся обратно
┌──────────────┐ ◄───────────────────────────────┘
│   Браузер    │   6. показывает страницу; интерактивные
│              │      кусочки ("use client") оживают
└──────────────┘      (меню, форма, переключатель темы)
```

**Ключевая мысль:** общая «рамка» сайта (шапка сверху, подвал снизу, мобильная плашка) описана **один раз** в `app/layout.tsx`. А меняется только серединка — `{children}` («дети», т.е. содержимое текущей страницы). Поэтому шапку/подвал не нужно копировать в каждую страницу.

<a name="6"></a>
## 6. Диаграмма: как файлы вложены друг в друга

Это «дерево вложенности» — что во что вставляется, когда строится страница. Сверху самый внешний слой, внутри — что в него вкладывается.

```
app/layout.tsx  (ОБЩАЯ РАМКА — одна на весь сайт)
│
├─ <head>: маленький скрипт, чтобы тема (свет/тьма) применялась
│          мгновенно, без "моргания"
│
├─ <SiteHeader>   ← components/layout/site-header.tsx
│   ├─ <Logo>          ← components/layout/logo.tsx
│   ├─ меню            ← берёт пункты из lib/site.ts (mainNav)
│   ├─ <ThemeToggle>   ← components/theme-toggle.tsx
│   └─ <Button>        ← components/ui/button.tsx (надпись из lib/site.ts → ctaCopy)
│
├─ <main> = { children }   ◄── СЮДА ВСТАВЛЯЕТСЯ ТЕКУЩАЯ СТРАНИЦА:
│   │
│   │   (оборачивается в app/template.tsx — анимация появления)
│   │
│   ├─ app/page.tsx ............... ГЛАВНАЯ
│   │   ├─ <HeroSystemDiagram>  (анимированная схема)
│   │   ├─ <Container>, <Reveal>, <Button>, <SectionHeader>
│   │   ├─ <FeatureGrid> → <TrustCard> / <ServiceCard>
│   │   ├─ <LocalCallout>, <CTASection>
│   │   └─ данные из content/home.ts, content/services.ts, content/process.ts
│   │
│   ├─ app/services/page.tsx ...... УСЛУГИ
│   │   ├─ <PageHero>
│   │   ├─ <ServiceAccordion> → берёт content/services.ts
│   │   └─ <CTASection>
│   │
│   ├─ app/process/page.tsx ....... ПРОЦЕСС
│   │   ├─ <PageHero>
│   │   ├─ <ProcessStep> (×5) → берёт content/process.ts
│   │   └─ <CTASection>
│   │
│   ├─ app/about/page.tsx ......... О НАС
│   │   ├─ <PageHero>
│   │   ├─ <FeatureGrid> → <TrustCard> → берёт content/beliefs.ts
│   │   ├─ <LocalCallout>
│   │   └─ <CTASection>
│   │
│   └─ app/contact/page.tsx ....... КОНТАКТЫ
│       ├─ <PageHero>
│       ├─ <ContactForm> ──► отправляет в app/api/contact/route.ts ──► письмо
│       │       └─ варианты полей из content/contact-options.ts
│       │       └─ проверка из lib/contact-schema.ts
│       └─ боковая панель (email, запись) из lib/site.ts
│
├─ <SiteFooter>   ← components/layout/site-footer.tsx
│   └─ ссылки из lib/site.ts (mainNav, footerServiceLinks), email
│
└─ <MobileCtaBar> ← components/layout/mobile-cta-bar.tsx (только на телефоне)
```

**Как это читать:** `<SiteHeader>` (шапка) содержит внутри `<Logo>`, меню и `<Button>`. А `<Button>` — это совсем маленький кирпичик из `components/ui/`. То есть кирпичики собираются из кирпичиков поменьше — как матрёшка. Самый низ матрёшки — это `ui/` (кнопка) и `content/` (тексты).

<a name="7"></a>
## 7. Главная идея архитектуры: 3 слоя

Если запомнить только одну вещь про устройство — запомни это. Весь сайт делится на **3 смысловых слоя**, и они зависят строго «сверху вниз»:

```
┌─────────────────────────────────────────────────────────────┐
│  СЛОЙ 3 — СТРАНИЦЫ (app/.../page.tsx)                         │
│  "Что показать и в каком порядке."                            │
│  Берёт данные из СЛОЯ 1 и собирает из кирпичиков СЛОЯ 2.      │
└─────────────────────────────────────────────────────────────┘
                      │ использует ▼
┌─────────────────────────────────────────────────────────────┐
│  СЛОЙ 2 — КОМПОНЕНТЫ / КИРПИЧИКИ (components/)                │
│  "Как это выглядит." Кнопки, карточки, шапка, форма, секции.  │
│  Не знают конкретных текстов — получают их через props.      │
└─────────────────────────────────────────────────────────────┘
                      │ наполняется данными из ▼
┌─────────────────────────────────────────────────────────────┐
│  СЛОЙ 1 — ДАННЫЕ И НАСТРОЙКИ (content/ + lib/site.ts)         │
│  "Какие тексты, услуги, пункты меню, email."                 │
│  Чистые данные, без оформления.                              │
└─────────────────────────────────────────────────────────────┘

   🎨 Сквозной слой ОФОРМЛЕНИЯ (везде сразу):
   app/globals.css  +  tailwind.config.ts  → цвета, шрифты, отступы
```

**Почему так удобно тебе как редактору:**
- Хочешь поменять **текст** → правишь только Слой 1 (`content/`), не трогая оформление.
- Хочешь поменять **внешний вид кнопки везде** → правишь один кирпичик в Слое 2 (`components/ui/button.tsx`).
- Хочешь поменять **цвет золота на всём сайте** → правишь одну строку в слое оформления (`app/globals.css`).

Это и есть то самое «переиспользование компонентов», про которое ты знаешь: текст и вид разделены, поэтому правка в одном месте применяется везде.

<a name="8"></a>
## 8. Диаграмма: как файлы зависят друг от друга

Здесь стрелка `A → B` читается как «**A использует (импортирует) B**». Внизу — фундамент, на который опирается всё.

```
                       app/layout.tsx
                  (рамка: шапка + подвал)
                  /         |          \
                 ▼          ▼           ▼
        site-header   [ page.tsx ]   site-footer
         (шапка)       любой из 5     (подвал)
            │          страниц            │
            │             │               │
            │             ▼               │
            │     components/sections/    │   ← крупные блоки:
            │   (page-hero, cta-section,  │     hero, секции, карточки,
            │    feature-grid, cards,     │     аккордеон, шаги процесса
            │    service-accordion, ...)  │
            │        /          \         │
            ▼       ▼            ▼         ▼
        ┌───────────────┐   ┌──────────────────────┐
        │ components/ui │   │     content/*.ts      │  ← ТЕКСТЫ и ДАННЫЕ
        │ (button,      │   │ (services, process,   │    (services, process,
        │  accordion)   │   │  home, beliefs,       │     home, beliefs,
        │ + motion/     │   │  contact-options)     │     contact-options)
        │ (reveal)      │   └──────────────────────┘
        └───────────────┘            │
                │                     ▼
                │              ┌──────────────┐
                └─────────────►│  lib/site.ts │ ← имя, email, меню,
                               │  lib/utils   │   тексты кнопок (ctaCopy)
                               └──────────────┘
                                      ▲
                                      │
        ФУНДАМЕНТ ОФОРМЛЕНИЯ (используется буквально всеми):
        app/globals.css  ←→  tailwind.config.ts
        (цвета/шрифты/отступы как переменные → классы Tailwind)


        ОТДЕЛЬНАЯ ВЕТКА — ФОРМА:
        contact/page.tsx → contact-form.tsx ─(отправка)→ app/api/contact/route.ts → 📧
                                  │                              │
                                  └──── оба проверяют ───────────┘
                                        lib/contact-schema.ts
                                        (+ content/contact-options.ts)
```

**Главные выводы из диаграммы:**
1. **Стрелки идут вниз.** Страницы зависят от компонентов, компоненты — от данных и оформления. Данные (`content/`) ни от чего «выше» не зависят — поэтому их безопаснее всего править.
2. **`content/` и `lib/site.ts` — это «дно» с текстами.** Самое частое место правок.
3. **`globals.css` + `tailwind.config.ts` — фундамент оформления.** Питают цветом/шрифтом вообще всё.
4. **Форма — единственная «двусторонняя» история:** браузер отправляет данные на сервер (`route.ts`), и обе стороны сверяются с одними и теми же правилами (`contact-schema.ts`).

<a name="10"></a>
## 10. Дизайн‑система: цвета, шрифты, отступы

Это самая «магическая» часть, поэтому разберём подробно.

### Как работают цвета (важно понять схему)

Цвета заданы **не** прямо в компонентах, а через **дизайн‑токены** (именованные переменные). Цепочка такая:

```
app/globals.css            tailwind.config.ts            компонент
──────────────────         ──────────────────            ──────────────────
--gold: 40 40% 34%   ──►   gold: hsl(var(--gold))  ──►   className="text-gold"
(задаём ЗНАЧЕНИЕ          (даём значению ИМЯ-класс)      (ИСПОЛЬЗУЕМ класс)
 цвета один раз)
```

1. В [`app/globals.css`](app/globals.css) задаётся **значение** цвета: `--gold: 40 40% 34%` (это золотой в формате HSL — тон/насыщенность/яркость).
2. В [`tailwind.config.ts`](tailwind.config.ts) это значение получает **имя‑класс**: `gold`.
3. В компоненте пишут короткий **класс**: `className="text-gold"` (золотой текст) или `bg-gold` (золотой фон).

**Вывод для тебя:** чтобы поменять оттенок золота на всём сайте — меняешь **одно число** в `globals.css`, и ничего больше.

### Две темы: светлая и тёмная

В `globals.css` два набора цветов:
- `:root { ... }` — **светлая** тема «Marble» (по умолчанию): тёплый белый фон, графитовый текст.
- `.dark { ... }` — **тёмная** тема «Graphite»: тёмно‑серый фон, кремовый текст.

Переключатель — кнопка `ThemeToggle` в шапке. Она просто добавляет/убирает класс `dark` у страницы, и весь сайт мгновенно перекрашивается. Выбор пользователя запоминается в браузере (localStorage), а маленький скрипт в `app/layout.tsx` применяет его до отрисовки, чтобы не было «моргания».

### Главные токены (что за что отвечает)

| Токен | Что это |
|---|---|
| `--background` | фон страницы |
| `--foreground` | основной цвет текста |
| `--muted` / `--muted-foreground` | приглушённые панели / второстепенный текст |
| `--card` | фон карточек |
| `--border` | цвет рамок |
| `--gold` | золотой акцент (текст, иконки) |
| `--primary` | главная кнопка (на свету — графит, в тёмной — золото) |
| `--gold-cta` | золотая кнопка на тёмных блоках |
| `--dark` | тёмные блоки (нижний призыв, центр схемы) |
| `--radius` | скругление углов (10px) |

### Шрифты
Подключены в [`app/layout.tsx`](app/layout.tsx):
- **Inter** — для обычного текста.
- **Plus Jakarta Sans** — для заголовков (`h1`–`h4`).

### Отступы и ширина
- `Container` (`components/layout/container.tsx`) центрирует контент. Бывает `narrow` (узкий), `default`, `wide` (широкий).
- Класс `.section` (в `globals.css`) задаёт одинаковые вертикальные отступы для всех больших блоков.

<a name="11"></a>
## 11. Форма обратной связи: полный путь письма

Это единственное место, где есть «настоящий бэкенд». Разберём по шагам.

```
1. Пользователь заполняет форму
   components/contact/contact-form.tsx  ("use client" — работает в браузере)
        │
        │  по ходу ввода проверяет данные правилами:
        │  lib/contact-schema.ts  (Zod: имя ≥2 символов, email валиден, ...)
        │  варианты кнопок (нужды/бюджет/сроки): content/contact-options.ts
        ▼
2. Нажатие "Request Free Audit" → данные уходят на сервер (POST‑запрос)
        │
        ▼
3. Сервер принимает: app/api/contact/route.ts
        │  - снова проверяет теми же правилами (нельзя доверять браузеру)
        │  - проверяет "ловушку для ботов" (honeypot — скрытое поле company_website)
        ▼
4. Что дальше — зависит от настройки EMAIL_PROVIDER:
        ├─ ничего не настроено → письмо просто пишется в консоль сервера
        │                        (удобно при разработке; форма "работает")
        ├─ "resend"   → отправка через сервис Resend (нужен RESEND_API_KEY)
        └─ "sendgrid" → отправка через SendGrid (нужен SENDGRID_API_KEY)
        ▼
5. Браузер показывает "Thanks — we got it." (или сообщение об ошибке)
```

**Зачем валидация в ДВУХ местах?** Проверка в браузере (`contact-form.tsx`) — для удобства пользователя (мгновенные подсказки). Проверка на сервере (`route.ts`) — для безопасности (браузеру нельзя доверять, данные могут подделать). Оба используют **один файл правил** `lib/contact-schema.ts`, поэтому правила не разъезжаются.

**Honeypot (медовая ловушка)** — скрытое от людей поле. Люди его не видят и не заполняют, а боты‑спамеры заполняют всё подряд. Если поле заполнено — заявка тихо отбрасывается.

**Сейчас письма реально не отправляются** — только пишутся в консоль. Чтобы включить настоящую почту, см. рецепт 12.7.

<a name="13"></a>
## 13. Как запустить сайт у себя на компьютере

> ⚙️ Особенность этого окружения (WSL): Node установлен локально в `~/.local/node`, поэтому перед командами нужно один раз указать путь. Это уже зафиксировано в памяти проекта.

```bash
# выполнять из папки проекта /home/bonzo/harborlinesystems

# 1) указать путь к Node (нужно в каждом новом терминале)
export PATH="$HOME/.local/node/bin:$PATH"

# 2) (только в первый раз) поставить библиотеки
npm install

# 3) запустить сайт для разработки — открой http://localhost:3000
npm run dev
```

Команды (из [`package.json`](package.json)):
| Команда | Что делает |
|---|---|
| `npm run dev` | запускает сайт локально, автоматически обновляет при правках |
| `npm run build` | собирает «боевую» версию (проверка, что всё компилируется) |
| `npm run start` | запускает уже собранную боевую версию |
| `npm run lint` | проверяет код на типичные ошибки |

**Рабочий цикл редактора:** запустил `npm run dev` → правишь файл → сохраняешь → смотришь изменения в браузере вживую.

**Публикация в интернет:** проще всего на Vercel (создатели Next.js). Подробности — в [`README.md`](README.md), раздел Deployment.
